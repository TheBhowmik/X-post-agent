Express.js is a Node.js framework that helps you:

1) Create a server

2) Handle HTTP requests (GET, POST, PUT, DELETE)

3) Build APIs easily

1️⃣ What is this package?

@modelcontextprotocol/sdk is the official SDK for Model Context Protocol (MCP).

👉 MCP is a standard way to let an AI (like Gemini, Claude, etc.)
talk to your code, call your tools, and use your server functions.

McpServer is a class that helps you:

1) Create an MCP-compatible server

2) Register tools (like addTwoNumbers, createPost)

3) Let AI discover & call those tools

SSEServerTransport enables an MCP server to communicate with AI clients using Server-Sent Events, allowing continuous server-to-client messaging over HTTP.

This enables SSE (Server-Sent Events)
Meaning: server can push messages continuously to the client (not one-time like REST).

1️⃣ First: what problem does this solve?

Your MCP server needs a way to talk to an AI client.

That “way of talking” is called a transport.

👉 SSEServerTransport is one such transport

2️⃣ What is SSE?

SSE = Server-Sent Events

📌 Meaning:

Server can continuously send data

Client stays connected

Communication is one-way (server → client)

3️⃣ Why MCP needs SSE

In MCP:

AI connects to your server

AI listens for:

tool results

responses

updates

SSE is perfect because:

No polling

No reconnecting again & again

Lightweight compared to WebSockets

4️⃣ What is SSEServerTransport?

SSEServerTransport is a class provided by the MCP SDK that:

Uses Server-Sent Events

Connects MCP server ↔ AI client

Works nicely with Express

So this line means:

“Import the class that lets my MCP server communicate using SSE.”

import express from "express";
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { SSEServerTransport } from "@modelcontextprotocol/sdk/server/sse.js";

const app = express();
const server = new McpServer({ name: "demo", version: "1.0.0" });

app.get("/sse", (req, res) => {
  const transport = new SSEServerTransport(req, res);
  server.connect(transport);
});

app.listen(3000);

What happens here?

Browser / AI hits /sse

SSE connection opens

MCP server attaches to it

AI can now call tools

SSE vs other transports (important)
Transport	   Direction	               Use
SSE	         Server → Client	         Web / HTTP
stdio	        Two-way	                 CLI tools
WebSocket	    Two-way	              Heavy realtime apps

z is an object that contains many helpers like:

z.string()

Each of these creates a schema.

5️⃣ Simple example (no MCP yet)

❌ If age is wrong:

server.tool(             //server.tool() means: “Register a tool that an AI can call.”
    "addTwoNumbers",     //1st argument: "addTwoNumbers" This is the tool name.AI uses this exact name to call the tool.Must be unique
    "Add two numbers",   //2nd argument: "Add two numbers" This is the description.  Helps AI understand when to use the tool  Natural language guidance  Better description = better AI decisions.
    {                    //3rd argument: input schema (Zod)   //This tells MCP:  “This tool requires two numbers: a and b.”  What happens:  1)AI sends arguments  2)Zod parses & validates  3)Invalid input → tool will NOT run ❌  This is critical for safety.
        a: z.number(),
        b: z.number()
    },
    async (arg) => {     //4th argument: the actual function. This is the tool implementation.  arg = validated input  Already safe to use  No need to re-check types
        const { a, b } = arg; //Destructuring the input. This means:  Extract a and b  From the validated input object  Equivalent to:  const a = arg.a; const b = arg.b;
        return {         //This is MCP’s response format.
            content: [   //content → array (can contain many messages) 
                {
                    type: "text",          // type: "text" → plain text output
                    text: `The sum of ${a} and ${b} is ${a + b}`
                }
            ]
        }
    }
)

This code registers an MCP tool named addTwoNumbers, validates its inputs using Zod, executes the logic,
and returns a structured text response to the AI.

const transports = {}; creates an object used to store and manage active transport connections, typically mapping client sessions to their SSE transports.

const transports = {};


👉 Why?

Many clients can connect

Each client gets a sessionId

2️⃣ Why is it called transports?

Because it will be used to store active transport connections.

In MCP (especially with SSE):

Each client connection has its own transport

You need to keep track of them

So you store them in an object

3️⃣ What will go INSIDE transports?

Later in code, you’ll see something like:

transports[sessionId] = transport;


So it becomes:

{
  "abc123": transportObject1,
  "xyz456": transportObject2
}


Each key:

Represents a connected client

Maps to its SSE transport

2️⃣ Creating the SSE transport

This line does a lot.

Creates a new SSE communication channel

3️⃣ Storing the transport

Every transport has a unique session ID.

Why this matters:

Multiple clients can connect

Each client needs its own transport

You must keep track of them

4️⃣ Handling client disconnect
res.on("close", () => {
    delete transports[ transport.sessionId ];
});


This runs when:

Client closes tab

AI disconnects

Network drops

What it does:

Removes the transport from memory

Prevents memory leaks

5️⃣ Connecting transport to MCP server

This is the final and most important step.

It means:

“Attach this communication pipe to my MCP server.”

After this:

MCP server knows about this client

2️⃣ What /messages does

This tells MCP:

“When the client wants to SEND something to me,

👉 Direction: Client → Server

3️⃣ Why SSE alone is not enough

SSE is one-way by design:

❌ Client cannot send data back on SSE

✅ Server can only push data

So MCP needs:

SSE → server ➜ client

HTTP POST → client ➜ server

That POST endpoint is /messages.

🔒 Final takeaway

/sse opens a live channel

SSEServerTransport = communication pipe

transports = active connections registry

server.connect() = enable MCP

SSE = listen only

/messages = talk back

Together = full duplex communication

MCP needs both

app.post("/messages", async (req, res) => {
    const sessionId = req.query.sessionId;
    const transport = transports[ sessionId ];
    if (transport) {
        await transport.handlePostMessage(req, res);
    } else {
        res.status(400).send('No transport found for sessionId');
    }
});
